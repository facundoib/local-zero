# F1 spike ‚Äî ingest-probe (2026-05-06)

Throwaway prototype run before committing F1 (Document ingestion) code into `src-tauri/`. Goal: de-risk two unknowns before paying the cost of wiring the deps into the main app.

## Questions the spike was meant to answer

1. Does `rusqlite` with the `bundled` feature build and link under SAC?
2. Does `pdf-extract` preserve Spanish characters (`√± √° √© √≠ √≥ √∫ √º ¬ø ¬°`) on real input?

## What was built

- `prototypes/ingest-probe/Cargo.toml` ‚Äî minimal binary depending on `rusqlite = { features = ["bundled"] }`, `pdf-extract`, `sha2`.
- `prototypes/ingest-probe/src/main.rs` ‚Äî runs an in-memory SQLite round-trip with Spanish text, then optionally runs `pdf-extract::extract_text_from_mem` on a path passed as argv[1] and inspects the output for mojibake (`√É¬±`/`√É¬°`/...) versus correctly preserved Spanish chars.

## What happened

Three build attempts (cargo build --release), each from a different shell lineage:

| # | Shell | Result |
|---|---|---|
| 1 | Bash (Claude Code) | `error: failed to run custom build command for num-traits v0.2.19` ‚Üí `os error 4551` (`ERROR_BLOCKED_BY_APP_CONTROL_POLICY`) |
| 2 | Bash, retry on warm `target/` | Same failure on the same `num-traits-*/build-script-build.exe` |
| 3 | PowerShell (`powershell.exe -NoProfile`) ‚Äî to verify lineage hypothesis | Same failure |

SAC blocks `build-script-build.exe` for `num-traits v0.2.19` deterministically. The block reproduces regardless of parent process (cargo, bash, powershell). This matches the already-documented behaviour in [`../../CLAUDE.md`](../../CLAUDE.md):

> `cargo clean && cargo build` | Bloqueado casi seguro (re-corre build scripts)

## What we learned

- **The spike's 4551 block is not specific to `rusqlite`.** It hit the first freshly-compiled build script in the dep graph (`num-traits`, pulled transitively by `pdf-extract` via `pom`/`postscript`/`euclid`). `rusqlite`'s C amalgamation never got to run, so we have no signal on it from this attempt.
- **Process lineage does not save freshly-built `build-script-build.exe`.** SAC's heuristic flags the fresh, untrusted binary itself; changing parent shell does not bypass it.
- **F1's dep set introduces several new build scripts to `src-tauri/Cargo.toml`** (`num-traits`, `libsqlite3-sys`, `pdf-extract` chain). Adding them na√Øvely will reproduce the same block in the main app, not just the prototype.

## What is still unverified

- pdf-extract Spanish character preservation on a real PDF.
- rusqlite bundled actually linking and the in-memory SQLite round-trip succeeding.
- Performance of either crate against the ¬ß11 budgets.

## Path forward (revised plan)

**Option A ‚Äî CI-based validation.** Add a GitHub Actions workflow that builds and runs `prototypes/ingest-probe` on a Linux/Windows runner (no SAC). The workflow takes a Spanish PDF (committed under `prototypes/ingest-probe/fixtures/`) and asserts the spike binary exits 0 with no mojibake. This closes the two unknowns without touching the local SAC posture. ~1h of setup.

**Option B ‚Äî Skip runtime spike, proceed to vertical slice.** Trust the published behaviour of `pdf-extract` (widely used, MIT, pure Rust) and `rusqlite` (de facto Rust SQLite binding). Wire F1 deps into `src-tauri/Cargo.toml` and accept that the first build will need a workaround (CI cache, exclusion, or one-time tolerance). Lower confidence, faster start.

**Option C ‚Äî Pre-build target on a non-SAC machine.** Maintainer (or CI) populates `target/` for `src-tauri` once with the F1 dep set, copies it back. Subsequent incremental builds work locally. Recovers full local dev loop at the cost of a one-time external build.

## Closure (2026-05-06, later same day)

**Spike PASSED end-to-end on the maintainer's local machine.**

After fixing a Rust source bug (byte-string literal with non-ASCII content; commit `21f0025`), `cargo clean && cargo build --release` succeeded locally ‚Äî meaning **all build scripts, including `num-traits`, ran without 4551**. This contradicts the SAC reproduction observed earlier in this session and forces a revision of the blocker model:

> The 4551 failures observed via Claude Code's Bash invocations were almost certainly a **process-lineage issue specific to that shell context**, not a deterministic SAC block on `num-traits` or `rusqlite`. From a normal user PowerShell, fresh build scripts pass. SAC discriminates on lineage, as documented; the maintainer's user shell is a trusted parent.

Implication: **F1 deps in `src-tauri/Cargo.toml` will build locally** when added by the maintainer. Plan C (pre-built `target/` from a non-SAC machine) is not needed. CI remains the secondary validation path.
