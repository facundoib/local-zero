## OQ#1 — Does Lemonade Server expose `/embeddings` for `bge-m3` as a managed recipe?

**Resolution:** Lemonade exposes `/v1/embeddings` as a first-class OpenAI-compatible endpoint. **`bge-m3` is NOT in the default registry**, but five other embedding models are bundled and ready to pull.

### Evidence

- The OpenAI-compatible API page lists `POST /v1/embeddings` as endpoint #3 of 13. ([source](https://lemonade-server.ai/docs/api/openai))
- `src/cpp/include/lemon/model_types.h` defines `EMBEDDING` as a first-class model type alongside LLM, RERANKING, AUDIO, IMAGE, TTS. The Tauri app ships a dedicated `EmbeddingPanel.tsx`. The router in `src/cpp/server/router.cpp` enforces NPU exclusivity per model type (LLM + audio + embedding can run concurrently on FLM/NPU).
- The default `server_models.json` ships **5 embedding model recipes**, all on the `llamacpp` backend:

| Model name | Checkpoint | Size | Notes |
|---|---|---|---|
| `nomic-embed-text-v1-GGUF` | `nomic-ai/nomic-embed-text-v1-GGUF:Q4_K_S` | 0.08 GB | English-leaning |
| `nomic-embed-text-v2-moe-GGUF` | `nomic-ai/nomic-embed-text-v2-moe-GGUF:Q8_0` | 0.51 GB | Multilingual MoE |
| `Qwen3-Embedding-0.6B-GGUF` | `Qwen/Qwen3-Embedding-0.6B-GGUF:...:Q8_0.gguf` | 0.64 GB | Multilingual, Qwen family |
| `Qwen3-Embedding-4B-GGUF` | `...Q8_0.gguf` | 4.28 GB | Multilingual, larger |
| `Qwen3-Embedding-8B-GGUF` | `...Q8_0.gguf` | 8.05 GB | Multilingual, top tier |

Source: [`src/cpp/resources/server_models.json` on main, scraped 2026-04-30](https://raw.githubusercontent.com/lemonade-sdk/lemonade/main/src/cpp/resources/server_models.json).

### Decision

**v0.1 default embedding model = `Qwen3-Embedding-0.6B-GGUF`.**

Rationale:
- Lemonade-managed recipe — zero install friction. User just runs `lemonade pull Qwen3-Embedding-0.6B-GGUF`.
- Multilingual including Spanish (Qwen3 family is multilingual-first).
- Same model family as our chat LLM (Qwen3 14B), which gives the tutorial narrative consistency ("misma familia de modelos para chat y embeddings").
- Q8_0 quantization (high precision; embedding quality is more sensitive to quantization than generation).
- Tiny (0.64 GB) — fits comfortably alongside the 14B chat LLM in any 16 GB VRAM budget.

### Implication for SPEC

- §5.3 Default models table: replace `bge-m3-GGUF` row with `Qwen3-Embedding-0.6B-GGUF` (0.64 GB Q8).
- F1/F2 functional spec: change references to "bge-m3" → "Qwen3-Embedding-0.6B-GGUF".
- §15 OQ#1: mark resolved.

## OQ#2 (light) — Is the chosen embedding model the right one for Spanish?

**Resolution:** `Qwen3-Embedding-0.6B-GGUF` is a defensible v0.1 default. Deeper benchmark eval deferred to v0.2.

### Evidence

- Multilingual MTEB benchmarks 2026 consistently rank Qwen3-Embedding-8B at the top among open-source multilingual models. The 0.6B variant inherits the same training recipe at lower capacity. ([source: KDnuggets MTEB review 2026](https://www.kdnuggets.com/top-5-embedding-models-for-your-rag-pipeline), [Cheney Zhang 2026 benchmark](https://zc277584121.github.io/rag/2026/03/20/embedding-models-benchmark-2026.html))
- BGE-M3 also performs strongly on multilingual MIRACL benchmark, with explicit Spanish coverage in the training data. Outperforms many proprietary services on multilingual retrieval. ([source: BentoML guide 2026](https://www.bentoml.com/blog/a-guide-to-open-source-embedding-models))
- Jina-embeddings-v3 is competitive at 8K context with 89+ languages, including Spanish. ([source: Jina](https://jina.ai/news/jina-embeddings-v3-a-frontier-multilingual-embedding-model/))

### Why Qwen3-Embedding-0.6B over BGE-M3 specifically

- **Operational**: BGE-M3 is not in Lemonade's default registry; we'd need to register it as custom (added install step). Qwen3-Embedding-0.6B is one `lemonade pull` away.
- **Narrative**: aligns with the chat LLM (also Qwen3) — coherent story for the video.
- **Quality**: 2026 benchmarks place Qwen3-Embedding-8B as top multilingual; the 0.6B inherits methodology.

### Decision

Qwen3-Embedding-0.6B-GGUF as primary. **Open follow-up for v0.2:** run a hand-labeled Spanish QA retrieval benchmark (~50 question-document pairs from real LATAM tech docs) comparing Qwen3-Embedding-0.6B, Qwen3-Embedding-4B, BGE-M3 (custom-registered), and `nomic-embed-text-v2-moe`. Pick winner for v0.2 default.

### Implication for SPEC

- §15 OQ#2: mark resolved for v0.1; track v0.2 follow-up as a v0.2 roadmap item in §14.

## OQ#4 — PDF text extraction crate

**Resolution:** Use `pdf-extract` for v0.1.

### Evidence

| Crate | License | Approach | Cross-platform | Pros | Cons |
|---|---|---|---|---|---|
| **pdf-extract** | MIT | Pure Rust | Yes | Simple API, single dep, popular | Edge cases on complex layouts |
| `lopdf` | MIT | Pure Rust, low-level | Yes | Maximum control | More work — chunk extraction not built-in |
| `pdfium-render` | Apache-2.0 | Rust binding to PDFium (C++) | Yes (with binary distro) | Best quality (Chrome's PDF engine) | Requires bundling/finding pdfium DLL — Tauri packaging complexity |
| `mupdf-rs` | AGPL-3.0 | Binding to MuPDF | — | High quality | **License RULED OUT** for our Apache-2.0 repo |
| `oxidize-pdf` | (TBD) | Pure Rust, has Spanish InvoiceExtractor | — | LATAM-aware feature | New (2026), unverified maturity |

Sources: [pdf-extract on crates.io](https://crates.io/crates/pdf-extract), [docs.rs](https://docs.rs/pdf-extract), [oxidize-pdf](https://crates.io/crates/oxidize-pdf/1.7.1).

### Decision

**`pdf-extract`** as v0.1 primary. Pure Rust = no Tauri bundling complications. UTF-8 / Spanish accents are handled by the underlying PDF text encoding rules; the crate exposes UTF-8 strings.

**Fallback if Spanish character handling fails in real testing:** swap to `pdfium-render` and accept the bundling complexity. Decision deferred to first integration test against a real Spanish CV PDF.

### Implication for SPEC

- F1: lock `pdf-extract` as the implementation choice.
- §15 OQ#4: mark resolved (with conditional fallback noted).

## OQ#6 — Streaming SSE in Tauri's `fetch`

**Resolution:** Use the WebView's native `fetch()` (NOT `@tauri-apps/plugin-http`) for SSE streaming from Lemonade.

### Evidence

- **Known bug**: `@tauri-apps/plugin-http` has documented streaming issues — the response loop doesn't iterate until the entire request completes ([tauri-apps/plugins-workspace#2415](https://github.com/tauri-apps/plugins-workspace/issues/2415)). This breaks `text/event-stream`.
- **Native fetch in WebView2** (Windows) and WKWebView (macOS) supports SSE / streaming responses fully — same as in any browser.
- Cross-origin: `localhost:13305` is cross-origin to the Tauri app's `tauri://localhost`. Lemonade's HTTP server is designed for local app integration and sends permissive CORS headers; native fetch should work without an intermediary.
- **Fallback**: if CORS becomes restrictive (e.g., Lemonade tightens defaults in a future release), use `tauri-plugin-cors-fetch` ([source](https://github.com/idootop/tauri-plugin-cors-fetch)) which explicitly supports response streaming and SSE through the Rust HTTP client.
- Avoid: `tauri-plugin-sse` (third-party, less maintained) and the official `plugin-http` (broken streaming).

### Decision

**Primary:** `window.fetch()` (native WebView API) with manual `text/event-stream` parsing. Do NOT add `@tauri-apps/plugin-http`.

**Fallback (if CORS issues arise):** add `tauri-plugin-cors-fetch`. No code changes beyond import path.

### Implication for SPEC

- F5: implementation note that streaming uses native fetch in the WebView, not the Tauri http plugin.
- §15 OQ#6: mark resolved.

## OQ#7 — Cold-start UX when Lemonade is NOT running

**Resolution:** Banner with "How to start Lemonade" link. No auto-start, no shell exec.

### Three options considered

| Option | Pros | Cons |
|---|---|---|
| **A. Banner with link** | Simple, transparent, no privilege escalation | User has to leave the app to fix |
| B. Auto-attempt `lemonade-server start` via shell | Best UX if it works | Requires Lemonade in `PATH`; failure modes are silent; security review needed for shell exec |
| C. Modal prompt "Start Lemonade?" with Yes/No | Middle ground | Extra friction; same shell exec risks as B |

### Decision

**Option A** for v0.1.

Rationale:
- Local Zero is a tutorial product. The user is *learning* what's underneath. Hiding Lemonade behind auto-start defeats the educational point — the user should see "ah, my local AI server is a separate process that I started".
- Shell exec from Tauri is a non-trivial security surface to vet for v0.1.
- The banner can include a short embedded explanation: "Local Zero usa Lemonade Server. Si no está corriendo, abrilo desde el menú Inicio o ejecutá `lemonade-server` en tu terminal."
- Defer auto-start to v0.2 once we have the security review and the install detection logic.

### Implication for SPEC

- F5: error path "Lemonade unreachable" → render banner component with troubleshooting link.
- §15 OQ#7: mark resolved.
