# F8 Redesign — From "Portfolio Piece" to Forced-Extension Starter

**Status:** Decided 2026-05-04
**Affects:** SPEC §1, §2 (goal #5), §7.1 (data model), §8 (F8 entire), §12 (acceptance #6), §14 (roadmap)
**Supersedes:** F8 in SPEC v0.0.1

## Problem

SPEC v0.0.1 specced F8 as a one-click export of a self-contained, runnable mini-project intended as "a portfolio-grade piece" the user pushes to GitHub. Every user who clicks Export receives the **same** 150-line `main.ts` plus the same README skeleton, package.json, sample placeholder document, and MIT license. The framing — *portfolio-grade* — is the load-bearing claim of the feature.

A 2026 deep research pass on what GitHub artifacts actually correlate with junior LATAM developer hires into AI / AI-Security roles found this framing unsupportable.

## Evidence summary (from research report §2)

Three findings drove the decision:

1. **Structural property of hire-correlated scaffolds.** Platforms whose users get hired off scaffold-derived artifacts (Frontend Mentor, CodeCrafters) all share one property: the user authors the *substantive code*. Frontend Mentor ships designs, not code. CodeCrafters ships a stage-gated test suite the user must satisfy with their own code. Tutorial scaffolds that ship finished implementations (Hartl Rails sample app and dozens of clones) exist on GitHub but are *not* the artifacts that got those juniors hired — those were the user's commits 50-200 *on top of* the scaffold. F8 v0.0.1 sits on the wrong side of this divide because the scaffold *is* the deliverable.

2. **Eval skills are the differentiator hiring managers cite.** Alexey Grigorev's analysis of 1,765 AI-engineering job descriptions (Q4 2025 / Q1 2026) finds 39.6% of AI-First roles explicitly require evaluation skills — labeled "the differentiator" in his guide. 40%+ of take-home assignments in the same dataset *are* RAG. Combined: arriving at an interview with a generic, eval-less RAG demo on GitHub is bringing the take-home to the take-home. Without evals it is table stakes.

3. **AI-slop crisis at OSS-maintainer level is documented and quantified.** cURL killed its bug bounty in early 2026 after the confirmed-vulnerability rate fell from "north of 15%" historically to "below 5%" in 2025 (Stenberg, primary source). The peakoss/anti-slop GitHub Action ships 34 detection rules across 57 options and reports it would have closed ~98% of slop PRs in early testing. tldraw closed external contributions late 2025. Identical scaffolding across many users is a known telltale of templated artifacts. Extrapolation to portfolio repos at scale is plausible (medium confidence — no callback-rate study isolates the signal cleanly), but the direction of pressure is unambiguous.

Counter-evidence considered and rejected: `create-react-app`, `cargo new`, `gh repo create` templates are not penalized — but they are universally understood as *first commits*, not deliverables. Their READMEs describe the user's app on top, not the scaffold. F8 v0.0.1 inverts that property.

## Decision

**Ship F8 in v0.1 as a forced-extension starter (Option B from the research report).** Remove the "portfolio-grade" claim from SPEC entirely; reframe F8 as a runnable starter the user must extend with their own corpus and evaluations before it is meaningful as a portfolio artifact.

Three gates must pass before the export button is enabled:

- **G1 — User-supplied corpus.** SHA-256 of at least one document in the active set must differ from the placeholder shipped with the app. Verifiable by hash.
- **G2 — User-authored evaluations.** ≥5 evaluation cases (`{question, expected_substring}`) authored by the user in an in-app editor before export. v0.1 uses substring matching only; LLM-as-judge deferred to v0.2.
- **G3 — User-authored README context.** Three free-text fields (>30 chars each), filled in the export dialog, injected verbatim into the exported README: "¿Qué problema resuelve?", "¿Por qué este dominio?", "¿Qué aprendiste?".

The exported repo additionally ships an `eval-runner.ts` (~30 lines) that loads the JSON eval cases and reports pass/fail. The post-export modal explicitly tells the user the repo is *not yet* a portfolio piece — what makes it one is the commits they make on top.

## Trade-offs

**Cost:** F8 dev estimate moves from ~2 days (v0.0.1) to ~3-5 days (Option B). On a v0.1 with 9 features this is non-trivial but absorbable.

**Friction:** Three gates raise abandonment risk. Mitigation: gate UX must be supportive — tooltips that explain what's missing, pre-filled examples in the eval editor, English-only fallback if user declines bilingual README.

**Bypassability:** A determined user can satisfy G1 by uploading any non-placeholder document, G2 by writing 5 throwaway evals, G3 by writing 31-char nonsense in each field. Accepted: gates raise the cost of producing a clone repo above zero, which is the structural property that matters. Perfect adversarial robustness is not the goal.

**Roadmap implication:** Option D (pre-wired differentiator add-ons — eval harness via Promptfoo/DeepTeam, LATAM corpus loaders, smoke red-team via Garak) goes to v0.2. Listed in §14.

## What was rejected

- **Option A (keep as-is, framing-only changes).** Does not change the substantive code, so does not move the artifact across the structural divide. The "portfolio-grade" claim cannot be defended on 2026 evidence regardless of how the README is worded.
- **Option C (replace F8 with an OSS contribution path).** Higher signal but ~25-60 dev days, multiple per-project toolchain integrations, and a bad failure mode (an embarrassing AI-slop PR sent to LangChain in the user's name). Not v0.1.
- **Cutting F8 entirely.** ~0.5 days to remove. Loses one of the two product reasons-of-being. Acceptable fallback if Option B implementation slips, not the preferred path.

## What would reverse this decision

- A peer-reviewed or instrumented study showing identical-scaffold portfolio repos do not penalize callback rates → revert toward Option A.
- Maintainer-side AI-slop tooling matures enough that newcomer OSS PRs become reliably accepted again → reconsider Option C for v0.2.
- Field signal that even forced-extension scaffolds are pattern-recognized and named in recruiter forums (watch r/devsarg, Carlos Azaustre / Midudev community posts in the 12 months after launch) → cut F8 entirely.

## Implementation notes for the implementer

- Add an `evals` table to the SQLite schema (§7.1): `(id, question, expected_substring, created_at)`. Scoped to the active corpus implicitly (per non-goal #1: one document collection at a time).
- The export button's enabled/disabled state is a derived predicate over (G1 & G2 & G3). Compute reactively in the frontend; do not duplicate in Rust unless the Rust-side commands also enforce it (defense in depth).
- The README template has three injection points for G3 strings; rendering should preserve user line breaks and trim leading/trailing whitespace.
- Acceptance criterion §12 #6 is rewritten to verify the gate behavior end-to-end.
